   212→D. MCP INTEGRATION ARCHITECTURE
   213→================================
   214→
   215→The MCP (Model Context Protocol) access layer is defined across three files:
   216→
   217→1. bridge/mcpaauth.py — RBAC + API key authentication
   218→   - 4 roles: ADMIN / OPERATOR / READONLY / GUEST
   219→   - 6 AuthResults: OK / INVALID_KEY / EXPIRED / RATE_LIMITED / FORBIDDEN / NOT_FOUND
   220→   - APIKey dataclass: key_id, key_value, role, agent_id, created_at, expires_at, rate_limit (declared but UNUSED), request_count
   221→   - Permission map (method-based): tasks/submit->tasks:write, tasks/status->tasks:read, vault/write->vault:write, vault/read->vault:read, a2a/agent-card->a2a:read
   222→   - Role permissions (with wildcard support): ADMIN={*}, OPERATOR={tasks:*, vault:*, a2a:*}, READONLY={tasks:read, vault:read, status:*}, GUEST={status:read}
   223→   - 3 default keys hardcoded: admin-001, operator-001, readonly-001
   224→   - Singleton via get_mcp_auth() with double-checked locking (threading.Lock)
   225→   - GAP: rate_limit field is declared but authenticate() never enforces it
   226→   - GAP: mcpaauth.py is NOT imported by bridge/server.py — two parallel auth systems exist
   227→
   228→2. bridge/server.py — JSON-RPC 2.0 governance server
   229→   - BridgeServer wraps FastAPI via create_app()
   230→   - 5 methods: tasks/submit, tasks/status, vault/read, vault/write, a2a/agent-card (A2A v1.1)
   231→   - Endpoints: POST /tasks/submit, POST /tasks/status, POST /vault/read, POST /vault/write, POST / (JSON-RPC router), GET /health
   232→   - Auth: HMAC-SHA256 signature via X-Nexus-Signature header (uses SecretStore, NOT MCPAuth API keys)
   233→   - Authz: KAIJU 4-variable (scope x clearance, impact x clearance, intent x action) via NexusGovernor.check_access()
   234→   - Token guard: pre-check 1000-token budget -> 429 if exceeded; tracks input/output tokens via TokenGuard; non-blocking on failure
   235→   - JSON-RPC errors: -32700 ParseError, -32600 Method Not Allowed, -32601 Unknown Method, -32602 Invalid Params, -32603 Internal Error; HeldError -> HTTP 202 with hold_ticket
   236→   - Response envelope: {jsonrpc: "2.0", result, trace_id, x-nexus-input-tokens, x-nexus-output-tokens}
   237→
   238→3. bridge/sdk.py — NexusClient client SDK
   239→   - HMAC-SHA256 request signing: signature = HMAC-SHA256(secret, "{secret}:{trace_id}:{payload}")
   240→   - X-Nexus-* headers: Agent-ID, Project-ID, Trace-ID, Signature, Lineage-ID (optional)
   241→   - RetryPolicy: max_retries=3, base_delay=1.0s, max_delay=30s, backoff_factor=2.0, retryable_codes=[429, 500, 502, 503, 504]
   242→   - CircuitBreaker: states CLOSED/OPEN/HALF_OPEN, failure_threshold=5, recovery_timeout=60s, thread-safe via threading.Lock
   243→   - Public API: submit_task, query_status, read_memory, write_memory
   244→   - Uses urllib (not aiohttp) — sync-only by default; async mentioned as optional in docstring
   245→
   246→HOW AN EXTERNAL MCP SYSTEM WOULD CONNECT:
   247→1. Acquire an agent_id + secret from SecretStore (env var NEXUS_SECRET_{AGENT_ID} or secrets.json file or master-key derivation)
   248→2. Register an API key with MCPAuth.register_key() — currently optional since server.py doesn't enforce it
   249→3. POST to http://bridge-host:7352/ with body {"method": "tasks/submit", "description": "...", "context": {...}, "kaiju": {scope, intent, impact, clearance}}
   250→4. Headers required: X-Nexus-Agent-ID, X-Nexus-Project-ID, X-Nexus-Trace-ID, X-Nexus-Signature, Content-Type: application/json
   251→5. Response: JSON-RPC 2.0 envelope with task_id + status + token_usage
   252→
   253→A2A v1.1: query via POST / with method="a2a/agent-card" returns AgentCard with capabilities, trust_band, negotiation_policies, status. NOTE: current _exec_agent_card() returns HARDCODED capabilities (code_generation, swarm_orchestration, governance_audit, memory_query, skill_discovery) — the docstring says "In production, this queries the Vault/Governor for dynamic, verified capabilities" but that wiring is not implemented.

   274→   - CircuitBreaker: threading.Lock() around state mutations
   275→   - Foreman: threading.Lock() around worker registry
   276→   - MCPAuth: threading.RLock() around key registry

   282→5. FORGE — DECLARATIVE TEAM DESIGN (Qualixar-inspired, arXiv:2604.06392):
   283→   - ForgeLoader.load_file(path) / load_string(yaml) -> TeamSpec
   284→   - TeamSpec: team_name + List[AgentSpec] + List[WorkflowStep] + metadata
   285→   - AgentSpec: role, model, traits, clearance, agent_id (resolved at runtime)
   286→   - WorkflowStep: step_number, agent_role, action, output_to (next|result), input_from (previous|step#)

   288→   - WorkflowStep.build_context() threads previous step results into next step's context

   291→6. SKILL AUTO-DISCOVERY (SkillSmith):
   292→   - skillsmith.py is a COMPAT SHIM — canonical impl lives in skill_smith.py
   293→   - Auto-forges SkillRecord after min_executions (default 3) tasks with success_rate >= success_threshold (default 0.85)
   294→   - Pattern extraction: takes first 2 words of prompt -> ^{word1} {word2}.* regex
   295→   - HermesRouter.route() checks skills first as fast-path (bypasses GMR selection if matched_skill found)
   296→   - SkillRecord stored in skill_registry dict + _task_history tracks per-(task_type, model) success history
   297→
   298→7. BAYESIAN SMOOTHING (ExperienceScorer):
   299→   - PRIOR_SUCCESS=10, PRIOR_FAILURE=2 prevent small-sample overfitting
   300→   - score = (PRIOR_SUCCESS + successes) / (PRIOR_SUCCESS + successes + PRIOR_FAILURE + failures)
   301→   - Falls back to model.quality_score (default 0.5) when no DB data
   302→   - record_outcome() upserts into model_performance table (model_id, task_class, success_count, failure_count, total_latency_ms)
   303→
   304→8. COST-AWARE MODEL SELECTION (CostOptimizer):
   305→   - Trivial/Standard complexity -> cost-optimized (cheapest above quality_threshold=0.5)
   306→   - Critical complexity -> prefer local models if available above threshold, else quality-first
   307→   - Default -> highest score
   308→   - Fallback: if no candidates above threshold, use highest-score model with "below threshold" reason
   309→
   310→9. DAG-BASED TASK ROUTING (EngineRouter):
   311→   - Tasks table: task_id, project_id, description, status, priority, context (JSON), heartbeat, timestamps
   312→   - task_dependencies table: parent_task_id -> child_task_id
   313→   - get_ready_tasks(): tasks with no deps OR all parent deps completed, ordered by priority DESC + created_at ASC
   314→   - DFS-based cycle detection (_would_create_cycle) before adding dependencies — raises ValueError on cycle
   315→   - Self-loop detection: parent_task_id == child_task_id rejected
   316→   - Status states: PENDING, IN_PROGRESS, COMPLETED, FAILED, BLOCKED, CANCELLED

   534→C. TRUST ENGINE v2.2 DEEP DIVE
   535→==============================
   536→
   537→PYTHON CANONICAL (nexus_os/governor/trust_engine_v2.py — TrustEngineV2 class):
   538→- Constants: BASELINE_SCORE=25.0, MAX_SCORE=99.5 (asymptotic plateau — never reaches 100), MIN_SCORE=0.0, SUCCESS_BASE_DELTA=4.0, FAILURE_DELTA=-10.0, CRITICAL_DELTA=-20.0, LOGISTIC_CENTER=50.0, LOGISTIC_STEEPNESS=10.0, BASE_DECAY_LAMBDA=0.02, CONVERGENCE_THRESHOLD=0.5, CDR_ESCALATION_TRUST=30.0, CDR_COLLAPSE_TRUST=15.0, MAX_HISTORY_LENGTH=100
   539→- 5-STEP UPDATE PIPELINE (update_trust method, lines 240-349):
   540→  1. ADAPTIVE TEMPORAL DECAY: lambda(t) = base_lambda * (1 + validator_disagreement_rate). decay_amount = lambda * (score - baseline). new_score = max(baseline, min(MAX_SCORE, score - decay)). Decay NEVER pulls below baseline (only excess decays).
   541→  2. DELTA CALCULATION: if !success: delta = -10 (FAILURE_DELTA), regression_events++. else: delta = SUCCESS_BASE_DELTA * logistic_scale(score, difficulty). logistic_scale: 1/(1+exp(-(T-50)/10)) * difficulty. At T=25 (baseline), logistic = 1/(1+exp(2.5)) ≈ 0.075 → success gain ≈ 0.3 points. At T=50, logistic = 0.5 → gain = 2 points. At T=90, logistic = 1/(1+exp(-4)) ≈ 0.98 → gain = 3.92 points. **WAIT — this means gains are SMALL at low trust and LARGER at high trust, contradicting the docstring "logistic scaling prevents gaming"**. The actual behavior is the opposite: low-trust agents barely recover, high-trust agents accelerate. The "anti-gaming" framing is questionable.
   542→  3. NON-COMPENSATORY CRITICAL: if danger == CRITICAL: delta = -20 (CRITICAL_DELTA, hard block), regression_events++, force cdr_stage to at least CASCADE. This overrides any success — even a successful CRITICAL action reduces trust by 20. This is the "non-compensatory" property.
   543→  4. APPLY DELTA + TRACK PEAK: new_score = max(MIN, min(MAX, score+delta)). Update peak_score. Append delta to history_delta (kept at last 100).
   544→  5. CDR STATE MACHINE (_update_cdr_stage): if score < 15 → COLLAPSE. Else if should_escalate(current_stage, score, regressions) → next_stage(). should_escalate rules: NORMAL+score<30 → DEGRADED; DEGRADED+regressions>=3 → MEMORY_CORRUPTION; MEMORY_CORRUPTION+score<20 → HALLUCINATION; HALLUCINATION+regressions>=5 → CASCADE. Recovery: if score>50 AND cdr_stage>NORMAL AND regression_events==0 AND convergence_turns>5 → step down one stage.
   545→- VELOCITY CALC (_calculate_velocity): weighted average of last 10 history_delta values, weights 1..10 (more recent = higher weight). Returns absolute value.
   546→- VAULT PERSISTENCE: store_track(agent_id, lane, "trust", "current", payload) where payload = {score, last_updated, convergence_turns, regression_events, cdr_stage, validator_disagreement_rate, total_validations, peak_score, history_delta[-20:]}. NOTE: "current" is a SINGLE KEY per agent:lane — overwrites prior state. Only last 20 deltas persisted. So the vault is NOT a chain — it's a current-state cache.
   547→- TRUST MATRIX OUTPUT: agent_id, lane, trust (rounded 2dp), cdr_stage, cdr_severity, convergence_turns, regression_events, total_validations, peak_score, trust_velocity.
   548→
   549→CDR 6 STAGES (CDRStage enum):
   550→  0=Normal, 1=Degraded Reasoning, 2=Memory Corruption, 3=Output Hallucination, 4=Cascade, 5=Collapse. Severity = index in order.
   551→
   552→DANGER 5 LEVELS (DangerLevel enum):
   553→  0=SAFE, 1=CAUTION, 2=RESTRICTED, 3=HIGH_RISK, 4=CRITICAL. CRITICAL is the only one with hard-block semantics.
   554→
   555→PARALLEL trust_scoring.py (HOT-PATH IMPL — 259 lines):
   556→- compute_score(inp, lane_params) — separate formula using LANE_PARAMS (qmin, n0, Rcrit, alpha, beta, gamma, eta, kappa, delta, epsilon) per lane (RESEARCH, REVIEW, AUDIT_SECURITY, COMPLIANCE, IMPLEMENTATION, ORCHESTRATION, SYNTHESIS).
   557→- Formula: q_gated = (Q - qmin) / (1 - qmin); Qeff = q_gated * (1 - exp(-n/n0)); P = alpha*U + gamma*D_plus - beta*R - eta*D_minus; raw_score = tanh(kappa * Qeff^delta * P); score = 0 if |raw_score| < epsilon else raw_score. Invariant: -1 <= score <= 1.
   558→- INVARIANTS: inv1 (blocked/unassigned/not_applicable → None), inv2 (hard_fail → -1 + ESCALATED), inv3 (score in [-1, 1]).
   559→- MemoryTracks class is in-process (no DB persistence). update_trust uses Beta-Binomial: alpha += Qeff*max(score,0); beta += Qeff*(max(-score,0) + 0.5*hard_fail). get_trust = alpha/(alpha+beta).
   560→- Authority band: >0.8 trust + strong capability = "elevated"; <0.3 or >5 failure patterns = "restricted"; else "standard".
   561→- THREE PARALLEL TRUST SYSTEMS:
   562→  1. trust_engine_v2.py (TrustEngineV2) — logistic + decay + CDR, persisted to vault
   563→  2. trust_scoring.py (compute_score + TrustScoringGate) — Beta-Binomial + lane params, in-memory only
   564→  3. /api/trust-engine/route.ts — dashboard computes lane trust with hard-coded `trust + modifier` heuristic, returns CDR via simple thresholds
