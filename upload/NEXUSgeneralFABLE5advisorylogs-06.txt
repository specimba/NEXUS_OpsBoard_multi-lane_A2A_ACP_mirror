## §12 The dispatch system — how deliverables move from this doc to agents

**Why packets exist.** A 2,200-line document is for you; an agent that receives
 all of it digs through forensics to find its three tasks — and demonstrably do
esn't (§10.5). So the doc is mechanically split: **one packet per executor** =
Part 0 verbatim (charter, board snapshot, write-surfaces, taxonomy, templates)
+ only that executor's cards + the permitted per-packet additions (repo path an
d isolation rules for SONNET5-CLI; inlined evidence and OG-2 outcomes for GLM52
, which cannot read host paths).

**Integrity.** Every packet header carries the master doc's sha256 (first 12),
DOC-VERSION, and generation timestamp. The agent's mandatory RECITATION echoes
the hash back — a stale or hand-edited packet is detectable in the first messag
e of the session. Packet board snapshots are **frozen at generation**; live sta
tus is OPS's board, updated from reports.

**The loop.** Dispatch (you paste a packet + "work NXM-0xx only") → RECITATION
(agent proves it can fill every field from its packet alone; if not, it files D
OC-DEFECT and stops) → execution under the card's gates → REPORT filed to `disp
atch\reports\` (or returned as the final message, for lanes without filesystem
reach — you relay those) → OPS intake (board flip, REV bump, new D-### transcri
ption, next-card dispatch). One SONNET5-CLI session per card, always — session
sprawl is how scope invasion happens.

**When something goes wrong.** BLOCKED report → the agent has already moved to
its next card; you decide whether to unblock (usually an OG or an elevation ste
p). DOC-DEFECT → OPS fixes the doc, bumps DOC-VERSION, regenerates the affected
 packet; agents never improvise around a defective order. Conflict with your li
ve instruction → you win, and the agent files NEW-FACTS so the doc catches up.

**Distribution order** lives in `dispatch\00_DISTRIBUTION_ORDER_2026-07-14.md`
— including each packet's "actionable when" condition (HERMES inert until NXM-0
21 + re-provisioning; GLM52 inert until OG-2; SONNET5-CLI repo cards inert unti
l their slice-commit edges clear).
